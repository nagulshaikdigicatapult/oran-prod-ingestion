O-RAN Production Ingestion Pipeline – Operational Runbook
1️⃣ Purpose

This runbook describes:

How to bootstrap on a fresh VM

How to run the ingestion pipeline end-to-end

Where each artifact comes from

What is committed vs generated

How to validate integrity

How to troubleshoot and recover safely

2️⃣ Repository Truth Model (Very Important)
🔐 Control-Plane (Committed to Git)

These files define deterministic behavior:

manifests/raw/manifest.json
→ Portal export (contains id + download_url)

inventory/id_filename_map.json
→ Stable mapping: { portal_id → official_filename }
→ Generated earlier via HTTP Content-Disposition join
→ Treated as a lockfile (DO NOT delete casually)

📦 Data-Plane (Generated Artifacts – NOT committed)

Generated by running the pipeline:

downloads/

extracted_flat/

extracted_docs/

downloads_by_title/

reports/

inventory/download_inventory.full.json

manifests/processed/normalized_manifest.json

These are reproducible and safe to delete.

6️⃣ ZIP Extraction Behavior

The pipeline:

Extracts each downloaded .zip into:
extracted_flat/<id>__<zip_stem>/

Recursively extracts nested zips

Deletes nested zips after extraction

Copies only .pdf/.docx/.xlsx into:
extracted_docs/

Invariant:

find extracted_flat -type f -iname "*.zip"
find extracted_docs -type f -iname "*.zip"

Expected:

0
0

🔟 id_filename_map.json – Where It Came From

This file was generated during the metadata-driven catalog phase by:

Taking each portal id

Calling the download endpoint

Reading the Content-Disposition header

Extracting the official filename

Writing { id → filename }

It is treated as a lockfile for deterministic idempotency.

11️⃣ Recovery Procedure

If corruption detected:

rm downloads/<corrupted_file>
python scripts/09_full_run_pipeline_v2.py

Only missing file will re-download.
